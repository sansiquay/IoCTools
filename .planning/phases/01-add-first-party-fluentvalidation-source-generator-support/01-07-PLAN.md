  <action>
    **ValidatorInspectCommand.cs:**
    - Follow existing command pattern from the CLI codebase
    - Command name: `validators` (or `validator-list` — check existing naming convention)
    - Functionality:
      1. Load project via MSBuild workspace (same as existing commands)
      2. Scan compilation for classes inheriting `AbstractValidator<T>` (by name, same detection as generator)
      3. For each validator found, display:
         - Validator class name (fully qualified)
         - Validated model type (the T)
         - Lifetime (from IoCTools attribute)
         - Whether it has composition edges (SetValidator/Include references)
      4. Table format output with colored lifetimes (green Singleton, blue Scoped, gray Transient)
      5. Support `--json` flag for machine-readable output
      6. Support `--filter` to filter by model type name

    **ValidatorGraphCommand.cs:**
    - Command name: `validator-graph`
    - Functionality:
      1. Load project and scan for validators (reuse detection logic)
      2. Build composition graph by parsing SetValidator/Include/SetInheritanceValidator
      3. Display graph as tree:
         ```
         OrderValidator [Scoped] -> Order
           └── AddressValidator [Scoped] -> Address (via SetValidator, injected)
           └── SharedRulesValidator [Scoped] -> Order (via Include, direct ⚠)
         ```
      4. Color-code: red for direct instantiation issues, standard lifetime colors
      5. Support `--why <ValidatorName>` flag (per D-19):
         - Trace the composition chain backwards to find what forces the lifetime
         - Output: "OrderValidator is Scoped because: it composes AddressValidator which depends on AppDbContext [Scoped]"
      6. Support `--json` for machine-readable graph output

    **Program.cs update:**
    - Register both new commands in the command dispatch
    - Follow the existing pattern for command registration

    Reuse FluentValidation type detection by name (same logic as generator). DO NOT take a dependency on the FluentValidation generator package — the CLI analyzes the Roslyn compilation directly using the same name-based patterns.
