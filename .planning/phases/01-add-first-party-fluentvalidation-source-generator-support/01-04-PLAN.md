    **CompositionGraphBuilder.cs:**
    - Namespace: `IoCTools.FluentValidation.Generator.CompositionGraph`
    - `internal static class CompositionGraphBuilder`
    - Method `BuildEdges(TypeDeclarationSyntax validatorDecl, SemanticModel model, string parentValidatorFqn)` returning `List<CompositionEdge>`:

    Walk `validatorDecl.DescendantNodes().OfType<InvocationExpressionSyntax>()` and for each invocation:

    1. **Extract method name:** Handle both `MemberAccessExpressionSyntax` (e.g., `RuleFor(...).SetValidator(...)`) and simple `IdentifierNameSyntax`. Get the method name string.

    2. **SetValidator detection:**
       - If method name is "SetValidator":
         - Get the first argument expression
         - If `ObjectCreationExpressionSyntax` (i.e., `new SomeValidator()`):
           - Get type info via `model.GetTypeInfo(creation).Type`
           - Create edge with `IsDirectInstantiation = true`, `CompositionType.SetValidator`
         - If `IdentifierNameSyntax` or `MemberAccessExpressionSyntax` (i.e., injected field/param):
           - Get symbol info via `model.GetSymbolInfo(expr)`
           - Resolve the type (field type, parameter type, or property type)
           - Create edge with `IsDirectInstantiation = false`, `CompositionType.SetValidator`
         - Set `Location` from the invocation's GetLocation()

    3. **Include detection:**
       - If method name is "Include":
         - Same pattern as SetValidator: check first argument for `new X()` vs injected reference
         - Create edge with `CompositionType.Include`

    4. **SetInheritanceValidator detection:**
       - If method name is "SetInheritanceValidator":
         - The argument is typically a lambda: `v => { v.Add<Dog>(new DogValidator()); }`
         - Walk the argument's `DescendantNodes()` for `InvocationExpressionSyntax` with method name "Add"
         - For each `.Add<T>()` call, extract the generic type argument from `GenericNameSyntax`
         - Check the argument to `Add<T>()` for `new X()` vs injected reference
         - Create edge with `CompositionType.SetInheritanceValidator` for each `.Add<T>()` found

    5. **Type resolution helper** (private method `ResolveChildValidatorType`):
       - Given an `ExpressionSyntax` argument, determine the child validator's fully-qualified type name
       - Handle: `ObjectCreationExpressionSyntax` -> `GetTypeInfo().Type`
       - Handle: `IdentifierNameSyntax` -> `GetSymbolInfo()` -> field/param/local symbol -> type
       - Return null if unresolvable (don't crash per generator constraint)

    6. **Guard:** Wrap all semantic model calls in null checks. If type resolution fails, skip the edge (don't crash). Per project constraint, generator never throws.

    **ValidatorPipeline.cs update:**
    - After collecting validators, add a composition graph building step
    - The composition graph edges should be built per-validator in the pipeline transform
    - Option A: Add `ImmutableArray<CompositionEdge> CompositionEdges` to ValidatorClassInfo
    - Option B: Build edges separately and combine in the emitter
    - Choose Option A for pipeline coherence. Update ValidatorClassInfo to include composition edges.
    - In the CreateSyntaxProvider transform, after constructing ValidatorClassInfo, call `CompositionGraphBuilder.BuildEdges(typeDecl, ctx.SemanticModel, fqn)` and pass the result to ValidatorClassInfo constructor.
    - This means ValidatorClassInfo needs an additional field. Update the model (from Plan 01) to include `ImmutableArray<CompositionEdge> CompositionEdges`.
    - Update IEquatable implementation to include CompositionEdges in equality (compare by SequenceEqual or count+content).
