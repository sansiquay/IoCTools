    **DirectInstantiationValidator.cs (D-13, D-14):**
    - Namespace: `IoCTools.FluentValidation.Diagnostics.Validators`
    - `internal static class DirectInstantiationValidator`
    - Method `Validate(ValidatorClassInfo validator, ImmutableArray<ValidatorClassInfo> allValidators, Action<Diagnostic> reportDiagnostic)`:
      1. Iterate `validator.CompositionEdges` where `edge.IsDirectInstantiation == true`
      2. For each direct instantiation edge:
         - Look up the child validator in `allValidators` by `ChildValidatorName`
         - If found (child is a discovered DI-managed validator):
           - Report IOC100 with message: `"{ChildValidatorTypeName} is directly instantiated via {CompositionType} but is a DI-managed service with lifetime {ChildLifetime}. Inject it through the constructor instead."`
           - Per D-14, if the child validator itself has DI dependencies (check child's ClassSymbol for [Inject] fields or constructor parameters that are services), append to message: `" {ChildValidatorTypeName} depends on {dep1}, {dep2} which won't be resolved."`
           - Use `edge.Location` for the diagnostic location
         - If not found in allValidators but the child type has [Inject] fields:
           - Also report IOC100 (child has DI dependencies even if not discovered as a validator)
           - Check child type's members for `InjectAttribute` by name
      3. Short-circuit pattern: if diagnostics disabled, return immediately

    **CompositionLifetimeValidator.cs (D-12):**
    - Namespace: `IoCTools.FluentValidation.Diagnostics.Validators`
    - `internal static class CompositionLifetimeValidator`
    - Method `Validate(ValidatorClassInfo validator, ImmutableArray<ValidatorClassInfo> allValidators, Action<Diagnostic> reportDiagnostic)`:
      1. Iterate `validator.CompositionEdges` where `edge.IsDirectInstantiation == false` (injected, not instantiated)
      2. For each injected edge:
         - Look up child validator in `allValidators` by `ChildValidatorName`
         - If found, compare lifetimes using standard hierarchy: Singleton > Scoped > Transient
         - Invalid: Singleton parent with Scoped/Transient child, Scoped parent with Transient child (if Transient holds state)
         - Actually, follow the STANDARD IoCTools pattern: a longer-lived service depending on a shorter-lived service is a captive dependency. Specifically:
           - Singleton parent + Scoped child = IOC101
           - Singleton parent + Transient child = IOC101
         - Report IOC101 with message: `"Validator {ParentName} ({ParentLifetime}) composes {ChildName} ({ChildLifetime}) via {CompositionType}. The child's shorter lifetime will be captured by the parent's longer lifetime."`
         - Use `edge.Location` for diagnostic location
      3. Short-circuit if diagnostics disabled
